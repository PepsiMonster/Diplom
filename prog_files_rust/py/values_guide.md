# Руководство по входным параметрам `experiment_values.py`

## 1. Зачем нужен этот файл

Файл `py/experiment_values.py` задаёт **внешние входные параметры** для имитационной модели.  
Чтобы поменять их для следующего эксперимента необходимо и достаточно изменить их в файле, после чего они:

1. проверяются на соответствие логике имитации в `values_validation.py`;
2. сохраняются в `py/generated/experiment_values.json` через `export_values.py`;
3. интерпретируются и валидируются внутри Rust-модуля `params.rs` используя `experiment_values.json`.

---

## 2. Какую систему описывает текущая модель

Текущая модель описывает **ресурсную систему массового обслуживания** с тремя типами ограничений:

- по числу заявок в системе;
- по числу одновременно обслуживаемых заявок;
- по совокупному объёму ресурса.

В систему поступает поток заявок, распределенный некоторым образом. Каждая заявка требует некоторый объём ресурса и прибор обслуживания. При этом скорость обслуживания может зависеть от текущего состояния системы. 

Имеются 2 конфигурации системы: `loss` и `buffer`, без очереди и с очередью. Поступающая заявка отклоняется, теряется, если все каналы обслуживания (или очередь в buffer конфигурации) заняты.

Также имеются 2 конфигурации state_dependent и constant для пременных `SERVICE_SPEED_PROFILE` и `ARRIVAL_RATE_PROFILE`. В state_dependent сценарии скорость обслуживания и прихода заявкок зависит от текущего состояния системы, а в constant задается константой.

## 3. Какие параметры сравниваются в рамках исследования

Ключевые параметры в рамках исследования: `WORKLOAD_FAMILY` и `ARRIVAL_PROCESS_FAMILY`

`WORKLOAD_FAMILY` задает распределение работы заявок в системе и может быть     "deterministic", "exponential", "erlang_2", "erlang_4", "erlang_8", "hyperexp_2", "hyperexp_heavy". 
Позволяет исследовать чувствительность к распределению времени обслуживания через отношение: 
- $T = \frac{W}{\sigma}$ в случае loss
- $\int_{0}^{T} \sigma_k(u) \,du \ge W$ в случае buffer

`ARRIVAL_PROCESS_FAMILY` задает распределение входящего потока заявок. По умолчанию задан экспоненциально; может быть задан "poisson", "erlang_2", "erlang_4", "hyperexp_2". 

Используя различные комбинации 2х семейств, получаем 3 сценария работы:
* __Чувствительность к распределению времени обслуживания__
    ```python
    WORKLOAD_FAMILY_PROFILE = "basic" или full
    ARRIVAL_PROCESS_FAMILY = ["poisson"] 
    ```
    Запуск:
    ```powershell
    python py/launcher.py --scenario-family workload-sensitivity
    ```

* __Чувствительность к типу входящего потока__
    ```python
    WORKLOAD_FAMILY_PROFILE = "fixed"
    ARRIVAL_PROCESS_FAMILY = ["poisson","erlang_2","erlang_4","hyperexp_2",]
    ```
    Запуск:
    ```poowershell
    python py/launcher.py --scenario-family arrival-sensitivity
    ```

* __Совместная чувствительность при обеих семействах__ 
    ```python
    WORKLOAD_FAMILY_PROFILE = "basic" или full
    ARRIVAL_PROCESS_FAMILY = ["poisson","erlang_2","erlang_4","hyperexp_2",]
    ```
    Запуск:
    ```powershell
    python py/launcher.py --scenario-family combined-sensitivity
    ```

Далее задаётся сама система: сколько заявок она вообще может держать, сколько из них может обслуживать одновременно и какой общий объём ресурса доступен.

## 4. Система как объект обслуживания

---

### `SERVERS_N`

`SERVERS_N` задаёт число обслуживающих приборов, то есть максимальное число заявок, которые могут обслуживаться параллельно. 
Чем больше `SERVERS_N`, тем легче системе “разбирать” поступившие заявки, не давая им накапливаться.

Увеличение `SERVERS_N` уменьшает долю отказов по ограничению приборов, снижает вероятность накопления очереди (для buffer), а пропускная способность возрастает(но в реальности обычно стоит денег).

Этот параметр следует выбирать совместно с `CAPACITY_K`. В текущей модели предполагается, что
- K = N для loss архитектуры
- K > N для buffer архитектуры

Соответсвующие проверки выполняются в `values_validation.py` 

### `CAPACITY_K`

Параметр `CAPACITY_K` задаёт **максимальное число заявок**, которые могут одновременно находиться в системе. Если система уже заполнена до уровня $K$, новый запрос не может быть принят только из-за ограничения по вместимости, даже если часть других ресурсов ещё не исчерпана.

Уменьшение `CAPACITY_K` повышает чувствительность к отказам по совместимости.

Данный параметр имеет значение в buffer сценарии; в сценарии loss K=N.
Также через `CAPACITY_K` задается длина очереди Q = K-N для buffer архитектуры.


### `TOTAL_RESOURCE_R`

Параметр `TOTAL_RESOURCE_R` задаёт суммарный объём ограниченного ресурса, который делится между всеми активными заявками. 

В прикладных интерпретациях  ресурс может означать: совокупный радиоресурс, вычислительный бюджет CPU, память(RAM), условный объём доступной полосы или энергетический ресурс.

Увеличение `TOTAL_RESOURCE_R` означает, что система может одновременно удерживать более ресурсоёмкие заявки. 

Содержательно `TOTAL_RESOURCE_R` особенно важен в тех сценариях, где требуется изучать не только факт обслуживания, но и конкуренцию заявок за общий ограниченный ресурс.

## 5. Входной поток заявок

Как было описано ранее, поток поступления описывается **профилем $\lambda_k$**, который при архитектуре loss $\lambda_k = \lambda$, а в архитектуре buffer зависит от текущего числа заявок $k$ в системе. Эффективная интенсивность поступления может ослабевать при приближении к насыщению, как в системах с admission control. 


### `ARRIVAL_NORMAL_VALUE`

Параметр `ARRIVAL_NORMAL_VALUE` задаёт интенсивность поступления в ненасыщенной зоне, то есть в тех состояниях, где система ещё далека от своего верхнего предела. 

Это основной параметр, определяющий, насколько агрессивен входной поток. Если он велик, система чаще находится в напряжённых состояниях и легче входит в режим насыщения. Если он мал, система работает в более спокойном режиме, и различия между сценариями обслуживания могут проявляться слабее.

`ARRIVAL_NORMAL_VALUE` имеет размерность “заявок на единицу модельного времени”.

Увеличение этого параметра переводит систему в режим высокой нагрузки. Уменьшение делает режим более лёгким и менее конфликтным по ресурсам.


### `ARRIVAL_THRESHOLD_OFFSET`

Параметр `ARRIVAL_THRESHOLD_OFFSET` не задаёт интенсивность сам по себе, а определяет момент, начиная с которого входной поток ослабляется. Порог вычисляется как
$$
k_{\text{thr}} = K - \texttt{ARRIVAL\_THRESHOLD\_OFFSET}.
$$
Актуален только для state_dependent сценариев.

Этот параметр контролирует, насколько рано модель начинает “реагировать” на приближение к насыщению. Если offset велик, снижение интенсивности начинается заранее, и система ведёт себя осторожнее. Если offset мал, поток дольше остаётся высоким, а система чаще оказывается в зоне жёсткой перегрузки.

С точки зрения прикладного смысла это параметр мягкого управления входной нагрузкой, влияет на то, как часто система будет сталкиваться с насыщением.


### `ARRIVAL_REDUCED_VALUE`

Параметр `ARRIVAL_REDUCED_VALUE` задаёт интенсивность поступления в перегруженной зоне, то есть после достижения порога $k_{\text{thr}}$. 

Если `ARRIVAL_REDUCED_VALUE` близок к `ARRIVAL_NORMAL_VALUE`, снижение интенсивности будет слабым и перегрузка продолжится. Если же он заметно меньше, то поток входящих заявок будет уменьшаться с повышением нагрузки, тем самым не давая системе полностью перегрузиться.

Например, люди в не будут заходить в супермаркет, если будут видеть большие очереди на кассе.


### `ARRIVAL_FULL_STATE_VALUE`

Параметр `ARRIVAL_FULL_STATE_VALUE` задаёт интенсивность поступления в полностью заполненном состоянии $k = K$. В большинстве сценариев разумным значением является `0.0`, так как система уже исчерпала свою вместимость.

Если оставить нулевое значение, это означает, что при полном состоянии новые поступления не генерируются как часть внутренней динамики интенсивности. Если задать положительное значение, модель будет формально учитывать попытки поступления и в полном состоянии.

Практически этот параметр чаще всего используется именно как фиксатор граничного условия: в полном состоянии приток новых заявок считается обнулённым или условием для отказа.

## 6. Заявка как носитель ресурса и работы

Каждая заявка в модели имеет две ключевые характеристики: сколько общего ресурса она занимает и какой объём обслуживания ей требуется. 
Ресурс отвечает за конкуренцию заявок между собой в системе, а объём работы — за продолжительность их пребывания под обслуживанием.
На данный момент ресурс никак не привязан к объему работы. 


### `RESOURCE_VALUES` и `RESOURCE_PROBABILITIES`

Пара `RESOURCE_VALUES` и `RESOURCE_PROBABILITIES` задаёт дискретное распределение требований заявки к ресурсу. Первый список описывает возможные объёмы потребляемого ресурса(сейчас [2,4,8,12,16]), второй — вероятности соответствующих значений(сейчас [0.3,0.25,0.25,0.15,0.05])

Увеличение этих параметров ведет к росту отказов по ресурсу, их уменьшение к снижению отказов по ресурсу. 


### `MEAN_WORKLOAD`

Параметр `MEAN_WORKLOAD` задаёт средний объём работы одной заявки. Он определяет, насколько “тяжёлой” является средняя заявка с точки зрения требуемого обслуживания. 
Не стоит забывать, что это среднее для разных распределений, т.е. его задание не фиксирует жестко сколько работы требует каждая заявка; среднее как ориентир.

Средняя трудоемкость задания. Если увеличить `MEAN_WORKLOAD`, заявки в среднем будут оставаться в системе дольше. Это приведёт к росту среднего числа заявок в системе, увеличению занятого ресурса и повышению вероятности отказа при прочих равных условиях.

## 7. Распределение объёма обслуживания

Помимо среднего объёма работы, существенна и **форма распределения**. Именно поэтому в модели отдельно задаётся семейство распределений обслуживания. Это напрямую связано с темой исследования: система может реагировать не только на средний объём работы, но и на вариативность распределения.

---

### `WORKLOAD_FAMILY`

Параметр `WORKLOAD_FAMILY` задаёт перечень распределений, которые будут сравниваться в экспериментальной серии. 

Предусмотрено 3 профиля `WORKLOAD_FAMILY` - full (7 распределений), basic (4 распределения), fixed (exponential). 
WORKLOAD_FAMILY_FULL = ["deterministic","exponential","erlang_2","erlang_4","erlang_8","hyperexp_2","hyperexp_heavy",]
WORKLOAD_FAMILY_BASIC = ["deterministic","exponential","erlang_4","hyperexp_heavy",]

Детерминированное распределение соответствует минимальной вариативности. Экспоненциальное представляет классическую случайную модель с заметной вариативностью. Эрланговские распределения позволяют получить более сглаженное поведение при том же среднем. Гиперэкспоненциальные распределения, наоборот, моделируют выраженную неоднородность и тяжёлый хвост.

### `WORKLOAD_HYPEREXP_P`

Параметр `WORKLOAD_HYPEREXP_P` задаёт вес быстрой ветви гиперэкспоненциального распределения. Внутри модели гиперэкспоненциальный сценарий интерпретируется как смесь двух экспоненциальных режимов: один отвечает за относительно лёгкие заявки, другой — за тяжёлые.

Если увеличить `WORKLOAD_HYPEREXP_P`, большая доля заявок будет приходиться на быструю ветвь. Однако при фиксированном среднем это означает, что меньшая доля оставшихся заявок будет ещё тяжелее. Поэтому изменение `p` влияет не только на состав смеси, но и на степень контраста между её компонентами.

### `WORKLOAD_HYPEREXP_FAST_MULTIPLIER`

Параметр `WORKLOAD_HYPEREXP_FAST_MULTIPLIER` определяет, насколько быстро обслуживается быстрая ветвь гиперэкспоненциальной смеси по отношению к базовому масштабу.

При его увеличении одна часть заявок становится ещё легче, а другая, при сохранении того же среднего, — ещё тяжелее. В результате усиливается неоднородность системы по времени обслуживания. 

### `WORKLOAD_HYPEREXP_HEAVY_P` и `WORKLOAD_HYPEREXP_HEAVY_FAST_MULTIPLIER`

Эти параметры задают более тяжёлый вариант гиперэкспоненциального сценария. Их роль аналогична предыдущей паре, но режим еще более контрастный.

## 8. Профиль скорости обслуживания $\sigma_k$

Профиль $\sigma_k$ описывает, как система справляется с нагрузкой в зависимости от текущего состояния. В текущей модели скорость обслуживания убывает по мере роста числа заявок, то есть перегруженная система обслуживает медленнее, чем свободная.


### `SERVICE_START_VALUE`

Параметр `SERVICE_START_VALUE` задаёт скорость обслуживания при малой загрузке. Это верхний уровень производительности системы, который наблюдается тогда, когда конкуренция за ресурс ещё невелика.

Если увеличить этот параметр, заявки будут обрабатываться быстрее, и система в целом станет устойчивее. Если уменьшить — система станет более “тяжёлой”, даже не меняя интенсивности поступления и среднего объёма работы.


### `SERVICE_STEP`

Параметр `SERVICE_STEP` определяет, насколько сильно убывает скорость обслуживания при росте числа заявок. Это параметр деградации.

Если параметр велик, производительность заметно падает при росте нагрузки, и чувствительность модели к распределению обслуживания резко усиливается. 


### `SERVICE_FLOOR_VALUE`

Параметр `SERVICE_FLOOR_VALUE` задаёт нижнюю границу скорости обслуживания. Даже если система сильно перегружена, скорость не падает ниже этого уровня.

## 9. Семейство прибытия заявок

### `ARRIVAL_PROCESS_FAMILY`

Параметр `ARRIVAL_PROCESS_FAMILY`  определяет семейство распределений входящего потока
ARRIVAL_PROCESS_FAMILY = [
    "poisson", # A∼Exp(λ)
    "erlang_2", # A∼Erlang(2, mean=1/λ)
    "erlang_4", # A∼Erlang(4, mean=1/λ)
    "hyperexp_2", # A∼HyperExp2​,E[A]=1/λ
]

Распределения внутри определяют скорость создания новых заявок.

Для гиперэкспоненциального распределения предусмотрены:
`ARRIVAL_HYPEREXP_P` задает вес быстрой ветви гиперэкспоненциального распределения,
`ARRIVAL_HYPEREXP_FAST_MULTIPLIER` определяет, насколько быстро создает заявки быстрая ветвь гиперэкспоненциальной смеси по отношению к базовому масштабу.

## 10. Временные параметры и серия прогонов


### `MAX_TIME`

Параметр `MAX_TIME` задаёт полную длину одного имитационного прогона. По умолчанию взят за 200, в тестовых прогонах использовался 50, прямпо пропорционален времени имитации. 

Этот параметр измеряется во внутреннем модельном времени, не в секундах.


### `WARMUP_TIME`

Параметр `WARMUP_TIME` задаёт длину начального интервала, который исключается из статистики. необходимо т.к. каждый прогон начинается из пустого состояния, а стартовая пустота не должна искажать оценки стационарного режима.

Задана как MAX_TIME*0.1.


### `REPLICATIONS`

Параметр `REPLICATIONS` задаёт число независимых повторов каждого сценария. Повторяется не один отдельный случайный путь как таковой, а **одна и та же конфигурация системы** с разными реализациями случайных величин. Прямо пропорционале времени выполнения имитации. 

При большом числе повторов оценки становятся устойчивее, а доверительные интервалы — уже.
Для отладки бралось 5 повторов. 


### `BASE_SEED`

Параметр `BASE_SEED` управляет воспроизводимостью случайных прогонов; одна и та же серия запусков может быть воспроизведена при повторном расчёте.

## 10. Метаданные эксперимента

### `SUITE_NAME`

Параметр `SUITE_NAME` задаёт имя серии экспериментов. Он используется для подписи артефактов, логов, таблиц и отчётов.

Этот параметр **не влияет на динамику модели и не должен интерпретироваться как физический режим системы**.

## 12. Как соотносить параметры с разными типами систем

Для радиодоступа 5G параметры можно трактовать как укрупнённые характеристики сектора или узла обслуживания: число одновременно допустимых сессий, число параллельно обслуживаемых потоков, общий объём доступного радиоресурса, требования сессии к ресурсу и объём передаваемой/обслуживаемой работы. Однако в той же форме модель может применяться и к edge-узлу, к вычислительному кластеру, к сервисному шлюзу или к другому объекту с ограниченными ресурсами.

---

## 13. Чего в модели нет

Текущая модель сознательно агрегирована. В ней не задаются явно геометрия сети, радиоканал, SINR, scheduler policy, mobility, handover, beamforming, numerology, HARQ, backhaul и многие другие механизмы, которые могут присутствовать в реальной 5G-системе. Эти эффекты не игнорируются полностью, но агрегируются в укрупнённые параметры потока, обслуживания и ресурса.

Это нужно помнить при интерпретации результатов. Цель модели состоит не в том, чтобы воспроизвести конкретную инженерную систему во всех деталях, а в том, чтобы исследовать, как ограниченная по ресурсам система реагирует на изменение распределения обслуживания и, в дальнейшем, типа входящего потока.

## 14. Где брать реальные ориентиры для калибровки под 5G

Если потребуется приблизить модель к реальной системе 5G, в качестве основных источников следует использовать стандартизованные документы по performance measurements и KPI. Спецификация 3GPP TS 28.552 посвящена измерениям производительности в 5G, а 3GPP TS 28.554 — end-to-end KPI для 5G и network slicing. Для O-RAN полезны O-RAN specifications и, в частности, документы, касающиеся O1 interface, так как они описывают management services, counters и взаимодействие с SMO.
